# DataOS 架构文档

> 对标 Alibaba DataWorks + Dataphin OneData + MES 数据集平台  
> 最后更新: 2026-06-09 (v2.0 — 对齐 MES 数据集平台架构)

---

## 一、项目定位

DataOS 是面向多项目数据接入、标准化建模与数据集生成的数据处理平台。作为 **RelOS（关系操作系统）的上游**，负责多源数据采集、清洗、标准化、建模和数据 API 服务化。

核心思路：以**公司级标准模型**为承载，以**字段字典和质量规则**为约束，以**项目级数据隔离**为边界，以**版本化数据集**为最终交付物。

```
DataOS（数据治理） → RelOS（关系推理） → AgentNexus（自然语言）
    上游                 中游               下游
```

## 三、双范围数据隔离模型

DataOS 采用**公司级共享 + 项目级隔离**的双层模型，对齐 MES 多客户场景：

```
公司级 (scope=global, 所有项目共享)          项目级 (scope=project, 项目隔离)
─────────────────────────────────          ─────────────────────────────

数据标准 (data_standards)                  数据源 (datasources)
  WORK_ORDER_CODE                            客户A MES库 → 项目3
  EMPLOYEE_NAME                              客户B MES库 → 项目2
  MRL_CODE                                  
                                           Pipeline (cleaning_pipelines)
编码字典 (code_dictionaries)                 用户数据清洗 → 项目3
  工单状态: 0→未发布, 1→已发布              订单预处理 → 项目2
                                          
                                          质量规则 (quality_rules)
字段映射 (field_mappings)                    工单号非空检查 → 项目3
  项目A: ORDER_NO → WORK_ORDER_CODE         
  项目B: OID → WORK_ORDER_CODE             业务过程 (business_processes)
                                            订单创建 → 项目3
主题域 (data_domains)                      
  生产调度、生产执行、质量管理               审计日志 (audit_logs)
                                            所有操作记录 → 项目级可追溯
```

| 层级 | 机制 | 实现 | 失败结果 |
|------|------|------|:---:|
| 路由层 | URL path project_id | FastAPI 路径参数自动解析 | 404 |
| 守卫层 | 用户身份 + 项目成员 + 角色级别 | `require_project_role()` 三重校验 | 403 |
| 数据库层 | FK 约束 + 唯一约束 | `project_id → projects.id` | IntegrityError |

### 数据分层存储

| 层 | 存储 | 格式 | 用途 |
|------|------|------|------|
| **Bronze** (原始) | MinIO | Parquet | 源系统数据完整副本，按 `projects/{pid}/datasources/{dsid}/` 分区 |
| **Silver** (清洗) | MinIO | Parquet | Pipeline 处理后数据，按 `projects/{pid}/pipelines/{name}/` 分区 |
| **Gold** (服务) | PostgreSQL | 表 | 最终可服务数据，Directus 自动生成 REST API |
| **元数据** | MySQL | 表 | 项目/用户/角色/标准/映射/字典/审计 |

## 六、RBAC 权限架构

```
用户最终权限 = 全局角色权限(user_roles) ∪ 项目角色权限(project_members)

全局角色 (平台级):          项目角色 (项目级):
  L6 super_admin              L4 project_owner
  L5 admin                    L3 project_admin
                              L2.5 developer
                              L2 operator
                              L1.5 editor
                              L1 viewer
```

| 能力 | 实现 |
|------|------|
| 角色体系 | 8 个系统角色 (3 全局 + 6 项目级) |
| 权限粒度 | 37 个 `resource:action` 权限点，8 类资源 |
| 权限计算 | 每次请求动态查询 → Redis 缓存 (15-30min TTL) |
| admin 穿透 | 全局 admin/super_admin 绕过所有项目成员检查 |
