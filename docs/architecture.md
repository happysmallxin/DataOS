## 一、项目定位

DataOS 是企业级数据治理平台，作为 **RelOS（关系操作系统）的上游**，负责多源数据采集、清洗、标准化和 API 服务化。RelOS 在其下游，接收标准化数据后进行关系图存储与推理。

```
DataOS（数据治理） → RelOS（关系推理） → AgentNexus（自然语言）
    上游                 中游               下游
```

### 数据源管理模块核心功能

| 功能 | 说明 |
|------|------|
| **多源注册** | 14 种数据源类型：6 种关系型/NoSQL + 3 种大数据 + 消息队列/搜索引擎/对象存储/API/爬虫/文件 |
| **配置加密** | Fernet 对称加密 (AES-128)，7 个敏感字段 (password/secret/token/access_key/private_key/api_key/connection_string) 入库前加密，API 返回脱敏为 `***` |
| **项目隔离** | `project_id` FK + `UNIQUE(project_id, name)` 约束 + API 层 `require_project_role` 守卫 |
| **连接管理** | 支持测试连接、数据同步、状态跟踪 (active/error/inactive) |
| **权限控制** | 6 级数据源权限：create/read/update/delete/test_connection/sync，按角色粒度分配 |
| **审计追溯** | 每次创建/删除自动记录 AuditLog (操作人/时间/目标/详情) |

项目管理是 DataOS 的**顶层组织单元和权限边界**。所有数据资产（数据源、爬虫、Pipeline、质量规则）都归属到项目下，实现资源隔离和权限控制。对标 DataWorks 工作空间 + Dataphin 项目中心。

```
                                    ▼
                          ┌──────────────────────────────┐
                          │        ProjectMember          │
                          │    用户-项目-角色 关联表        │
                          │    (project_id, user_id)      │
                          │    UNIQUE + role_id FK        │
                          └──────────────┬───────────────┘
                                         │
                    ┌────────────────────┼────────────────────┐
                    │                    │                    │
                    ▼                    ▼                    ▼
          ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
          │    Role      │    │  Permission  │    │  AuditLog    │
          │  8个系统角色  │    │  37个权限点   │    │  全量操作审计 │
          │ global/project│   │ resource:action│   │  project隔离  │
          └──────────────┘    └──────────────┘    └──────────────┘
```

### 7.2 三层数据隔离

```
请求 → [Layer 1: URL path project_id] → [Layer 2: require_project_role 守卫]
      → [Layer 3: DB FK 约束 + UNIQUE(project_id, name)]
```

| 层级 | 机制 | 实现 | 失败结果 |
|------|------|------|:---:|
| 路由层 | URL path 注入 project_id | FastAPI 路径参数自动解析 | 404 |
| 守卫层 | 用户身份 + 项目成员 + 角色级别 | `require_project_role()` 三重校验 | 403 |
| 数据库层 | FK 约束 + 唯一约束 | `project_id → projects.id` | IntegrityError |

### 7.3 RBAC 权限架构

```
用户最终权限 = 全局角色权限(user_roles) ∪ 项目角色权限(project_members)

全局角色 (平台级):          项目角色 (项目级):
  L6 super_admin              L4 project_owner
  L5 admin                    L3 project_admin
                              L2.5 developer
                              L2 operator
                              L1.5 editor
                              L1 viewer
```

| 能力 | 实现 |
|------|------|
| 角色体系 | 8 个系统角色 (3 全局 + 6 项目级)，对标 DataWorks 专业版 |
| 权限粒度 | 37 个 `resource:action` 权限点，8 类资源 |
| 权限计算 | 每次请求动态查询 DB → Redis 缓存 (15-30min TTL) |
| admin 穿透 | 全局 admin/super_admin 绕过所有项目成员检查 |
| 缓存失效 | 角色变更/成员增删时主动删除 Redis key |

### 7.6 关键设计决策

| # | 决策 | 对标来源 | 说明 |
|---|------|---------|------|
| 1 | 双范围 RBAC + 每次请求计算 | DataOS 原创 | 角色变更即时生效，DataWorks 登录时固化 |
| 2 | owner_id 不可变 | DataWorks | 仅记录创建者(审计)，权限走 project_members |
| 3 | admin 全局穿透 | DataWorks | 平台管理员不需要加入每个项目 |
| 4 | 8 角色拆分 developer/operator | DataWorks 专业版 | 开发运维分离 |
| 5 | Fernet 配置加密 + API 脱敏 | DataWorks KMS | 自建部署用 AES-128 |
| 6 | 全量审计日志 + before/after diff | DataLeap | 自建场景中更实用 |
| 7 | Redis 权限缓存 + 主动失效 | DataOS 原创 | 减少每次请求的 DB JOIN |
